## 验证一:PostgreSQL 10 全链路实测

**为什么选 PostgreSQL 10.23**:采购参数要求国产数据库"基于 PG 内核 ≥9.6(优先 ≥10)";瀚高 V5 与 PG10 内核同步,PG10 即参数最低档的最严格代理——在 PG10 上通过,意味着在任何更新的 PG 系国产库内核上方言兼容风险更低。

**环境**:本机 Windows x64;PostgreSQL 10.23(EDB 免安装 zip 版,端口 54310,initdb 全新实例);POC 副本 `poc-backend` 与生产代码同源。

**过程与结果**(每步都是真实执行,非纸面推演):

| 步骤 | 操作 | 结果 |
|---|---|---|
| 1 | `schema.prisma` 的 provider 由 `sqlite` 改为 `postgresql` | ✅ 仅此一行改动,兑现了 schema 头注释的承诺(模型早已避开 enum/array/native JSON) |
| 2 | `npx prisma generate` | ✅ 客户端生成成功 |
| 3 | `nest build` 生产构建 | ✅ |
| 4 | `prisma migrate dev` 全新生成并应用 PG 迁移 | ✅ **50 个模型一次性通过**,库内 51 张表(含迁移记录表) |
| 5 | `npm run db:seed` 全量种子 | ✅ 党组织 52 / 行政机构 53 / 用户 10 / 角色 6 / 权限 29 / 字典 9(70 项)/ 导航 / 示例展厅 / 会场 60 座次全部写入 |
| 6 | `node dist/src/main.js` 启动(PORT=3101) | ✅ 1 秒就绪,`/api/health` 返回 `database: "up"` |
| 7 | 12 项 API 冒烟(见下) | ✅ **12/12 PASS** |
| 8 | `psql` 直查 `"AuditLog"` | ✅ 登录 + 字典增删改的审计流水真实落在 PG 表中 |

**12 项冒烟明细**:健康检查 / 登录签发 JWT(含审计 INSERT)/ 用户画像(多表 join:组织归属+角色)/ 站点设置(JSON 列读取)/ 行政组织树(53 节点递归组装)/ 用户列表(关联查询)/ 字典列表 / 字典新建(INSERT+审计)/ 字典项 ×2(子表写入)/ 字典改名(UPDATE)/ 字典删除(级联 DELETE+回读 404)/ 考核计分引擎试算(score=4.5/6 精确)。

**顺带验证的部署事实**:PG10 Windows zip 版缺 VS2013 运行库(`msvcr120.dll`)导致 0xC0000135——这类"运行库坑"在麒麟上同样可能出现(glibc 版本),印证了把"到货实机验证"写进合同的必要性。

## 验证二:双平台离线部署包(海光 x86_64 + 鲲鹏 ARM64)

内网信创环境通常不能在线 `npm install`,部署包必须在有网机器上预先备齐全部平台二进制。实测结论:**可行**,四类原生组件双平台全部成功拉取并核实到具体文件:

| 组件 | x64(海光) | arm64(鲲鹏) |
|---|---|---|
| sharp 0.34.5(图像处理) | ✅ sharp-linux-x64.node + libvips 15.9MB | ✅ sharp-linux-arm64.node + libvips 16MB |
| @napi-rs/canvas 0.1.80(PDF 解析) | ✅ skia.linux-x64-gnu.node 29.8MB(glibc 变体) | ✅ skia.linux-arm64-gnu.node 24.5MB |
| Prisma 5.22 查询引擎 | ✅ rhel-openssl-1.1.x / 3.0.x(麒麟 V10 用 1.1.x) | ✅ linux-arm64-openssl-1.1.x / 3.0.x |
| Node.js 22 LTS 本体 | ✅ linux-x64.tar.xz 29.6MB(HTTP 200 实测) | ✅ linux-arm64.tar.xz 28.8MB |

**离线包体积**:x64 约 99MB、arm64 约 91MB(只带 OpenSSL 1.1.x 变体,平台无关的纯 JS 依赖另计、两平台通用)。

**实测挖出的两个坑(打包脚本必须处理,否则部署当天翻车)**:
1. **npm 跨平台拉取 sharp 必须四标志齐全**:`npm install --os=linux --cpu=x64 --libc=glibc --force --ignore-scripts sharp@0.34.5`。缺 `--libc=glibc` 时 npm 在 Windows 上**静默跳过** sharp 的 linux 平台包(不报错,装完是空的)。
2. **`prisma generate` 只带出查询引擎,不带 Linux 版 schema-engine**——目标服务器上跑 `prisma migrate deploy` 会尝试联网下载。解法(已实测 URL 全部 HTTP 200):从 `https://binaries.prisma.sh/all_commits/<engines-commit>/rhel-openssl-1.1.x/schema-engine.gz` 预取(arm64 同路径),或打包机上设 `PRISMA_CLI_BINARY_TARGETS` 环境变量后安装 prisma。

## 复现指引(POC 留档)

POC 全套环境在会话 scratchpad(`poc-backend/`、`pg10/`、`offline-deps-poc/`),核心复现序列:

```bash
# 1. 数据库:initdb -U postgres -E UTF8 --no-locale && pg_ctl -o "-p 54310" start && createdb djyy_poc
# 2. 后端副本:provider 改 postgresql;删 prisma/migrations(SQLite 方言);
export DATABASE_URL="postgresql://postgres@localhost:54310/djyy_poc"
npx prisma migrate dev --name init_postgres   # 全新生成 PG 迁移
npm run db:seed && npm run build
PORT=3101 node dist/src/main.js
node smoke.mjs                                # 12 项冒烟
```

对瀚高 / 金仓试用版:同序列,仅 DATABASE_URL 换成目标库(金仓需确认 PG 兼容模式 DB_MODE=pg,并可能需在 URL/驱动侧规避 version() 探测差异)——**这正是要求中标方配合完成、并作为付款前置条件的那次验证**。
